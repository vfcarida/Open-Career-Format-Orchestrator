# Supply Chain Security

This document describes the controls that secure the AKCP release pipeline and how consumers can verify the integrity of published artifacts.

---

## Implemented Controls

### 1. Pinned GitHub Actions (SHA-locked)

All GitHub Actions in AKCP workflows are pinned to specific commit SHAs rather than mutable version tags. This prevents upstream hijacking of workflow dependencies.

```yaml
# Example — pinned to exact SHA, not a floating tag
uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # v4.2.2
```

### 2. Frozen Lockfile

CI and release builds always use `pnpm install --frozen-lockfile`. This prevents unintended dependency resolution drift between runs.

### 3. Automated Scanning

| Control | Tool | Scope |
|---------|------|-------|
| Static analysis | **CodeQL** | PRs + scheduled |
| Dependency vulnerabilities | **Dependabot** | npm + GitHub Actions |
| PR vulnerability gate | **Dependency Review** | New PRs |
| Scorecard | **OpenSSF Scorecard** | Weekly |

### 4. Least-Privilege Workflows

Every workflow file declares an explicit `permissions:` block. Only the release workflow requests elevated rights (`contents: write`, `id-token: write`, `attestations: write`), and only for the jobs that require them. All other workflows default to `contents: read`.

### 5. Software Bill of Materials (SBOM)

AKCP generates an SPDX-JSON SBOM on every CI run and every release:

- **CI**: Generated by `anchore/sbom-action` (Syft), uploaded as a workflow artifact (`sbom-spdx`) retained for 30 days.
- **Release**: Generated and attached as `sbom.spdx.json` to every GitHub Release asset listing.

**Format**: [SPDX 2.3](https://spdx.github.io/spdx-spec/v2.3/) (JSON serialization)  
**Generator**: [Syft](https://github.com/anchore/syft) via `anchore/sbom-action`

### 6. Build Provenance Attestation

Every release attests the built `packages/*/dist/**` artifacts using **GitHub Artifact Attestations** (backed by Sigstore). This creates a cryptographically verifiable link between the published binary and the specific GitHub Actions run that produced it.

The attestation is stored in the GitHub attestation log and can be verified offline.

### 7. SBOM Attestation

The `sbom.spdx.json` file itself is separately attested, providing a verifiable chain: source → SBOM → binary artifacts.

### 8. NPM Provenance

Packages are published with `--provenance`, which attaches a Sigstore-backed provenance statement to each npm package. This allows any consumer to verify that a package was built from a specific commit in this repository.

## Verifying Release Artifacts

### Verify Build Attestation (GitHub CLI)

```bash
# Install GitHub CLI if not already installed
# https://cli.github.com/

# Verify a downloaded artifact
gh attestation verify <path/to/artifact> \
  --repo vfcarida/Agent-Knowledge-Compiler-and-Control-Plane

# Example: verify the CLI dist artifact
gh attestation verify packages/cli/dist/index.js \
  --repo vfcarida/Agent-Knowledge-Compiler-and-Control-Plane
```

### Verify NPM Package Provenance

```bash
# Verify the provenance of a published npm package
npm audit signatures

# Or check a specific package
npm install @akcp/cli@<version>
npm audit signatures
```

### Verify SBOM Integrity

```bash
# Download the sbom.spdx.json from the GitHub Release assets, then:
gh attestation verify sbom.spdx.json \
  --repo vfcarida/Agent-Knowledge-Compiler-and-Control-Plane
```

## SLSA Posture

> [!IMPORTANT]
> AKCP makes **no false SLSA level claims**. The current posture is documented honestly below.

| SLSA Requirement | Status | Notes |
|---|---|---|
| L1 — Build scripts checked in | ✅ | All build scripts in `.github/workflows/` |
| L1 — Build provenance available | ✅ | `actions/attest-build-provenance` on every release |
| L2 — Hosted build platform | ✅ | GitHub Actions (hosted runners) |
| L2 — Build provenance authenticated | ✅ | Sigstore-backed via GitHub OIDC |
| L2 — Hermetic build | ⚠️ Partial | `--frozen-lockfile` + `--ignore-scripts` on release, but network access remains |
| L3 — Isolated build | ❌ Not yet | Requires ephemeral, isolated runners (e.g., GitHub's hardened runners) |
| L3 — Tamper-evident log | ❌ Not yet | Requires Rekor log integration |

**Current level: SLSA L1 (provenance available) → approaching L2 (authenticated provenance).**

SLSA L3 is a roadmap target requiring additional infrastructure investment.
